# Composer Suggestions (tools + prompt enhancement)

> _As the user pauses while typing in chat, a non-modal popover above the composer suggests (a) the most relevant tools from the currently-selected mode/toolset — ranked locally with embeddings + the user's own usage history — (b) whole **extensions** that aren't wired into the conversation yet but match the draft's intent, and (c) a one-click prompt-enhancement rewrite generated by a small local model running in an optional Ollama sidecar. Everything runs on the operator's own hardware; drafts are never stored._

## Intent

Users don't know what the mode they picked can actually do, and first-draft prompts are often vague. Suggestions close both gaps at the moment of typing: tool chips teach the active toolset ("this mode can `scan`"), and the ✨ rewrite offers a sharper prompt without ever auto-applying. The architecture was validated by deep research + an adversarial review panel; the load-bearing verdicts are baked into the design:

- **Tool relevance is retrieval, not generation.** Production tool-routing consensus (Anthropic Tool Search, Bedrock AgentCore, MCP-scaling literature) uses embeddings/BM25 — an LLM pre-flight is slower, costlier, and no better. The tool half therefore needs **no sidecar, no GPU, no cold-start data**.
- **The generative half is strictly optional.** Prompt rewriting genuinely needs a M/LLM, so it rides a local model sidecar with graceful degradation: sidecar down ⇒ chips keep working, the ✨ row hides.
- **1B-class is the CPU default.** Measured CPU latency: 1B ≈1.5–3.5 s per short rewrite (viable on a pause), 4B thinking-off ≈5–10 s (marginal), 4B thinking-on ≈60 s (unusable). `qwen3:1.7b` (Apache-2.0, thinking suppressed) is the default; GPU hosts opt into `qwen3:4b` via one env var.
- **Measurement is a ship requirement.** Every impression/accept/dismiss lands in `suggestion_feedback` (content-free) so "does the enhancement earn its sidecar?" is answerable with data.

## How it works

### Tool half (embeddings, in-process)

1. `ChatInput.svelte` debounces typing pauses (`SUGGEST_DEBOUNCE_MS` = 600 ms; drafts under 12 normalized chars never fire) and POSTs `{draft, conversationId, modeId, include:["tools"]}` to `/api/composer/suggest`. `modeId` is always present and authoritative — same freshness semantics as `/api/tools?modeId=`.
2. The route resolves the runtime tool surface via `resolveScopedTools` (`web/src/lib/server/scoped-tools.ts`) — the SAME `computeModeToolScope` + `applyToolFilters` core the executor runs, extracted from `/api/tools`, so suggestions can never name a tool the mode wouldn't grant.
3. `src/suggest/intent-rank.ts` scores candidates: cosine(draft embedding, tool-description embedding), blended with a per-user prior. The draft embedding is **transient** (never persisted); tool-description embeddings come from the in-process MiniLM embedder (`src/memory/embeddings.ts`) through a content-keyed LRU (`src/suggest/embedding-cache.ts` — a changed description is a new key, so reindexing is automatic and lazy).
4. The prior (`src/suggest/user-tool-priors.ts`) is recency-decayed usage frequency from the user's own `tool_calls` rows (names + timestamps only) — the day-one "learns from your prompts and tool calls" signal. **Spam guard:** the relevance threshold applies to the RAW cosine, so a popular tool can only be boosted among relevant candidates, never ride its prior into an unrelated draft; a never-used tool with a strong match always surfaces (cold start).
5. Top-k (4) render as chips. Clicking a tool chip appends `![ext:<name>]` to the draft (the standard mention token — the existing inline-tool flow takes over from there); built-in chips are informational. Esc or × dismisses without re-nagging for the same draft.

**Warm-up (cold model).** The in-process MiniLM embedder takes ~2 min to initialize after a restart, and `generateEmbedding()` **blocks** on that load — so an ungated suggest request would hang for the whole window. Two things prevent that: (a) `startBackgroundTimers()` fire-and-forgets `warmupEmbeddings()` at boot (lazy-imported, never blocks boot) so the model is usually ready before the first pause; and (b) if a request still lands while warming — `isEmbeddingReady()` is false, or the embedder throws mid-flight — the route ranks **lexical-only** (token overlap; the same `0.28`/`0.35` relevance gates keep junk out, and authored `suggestExamples` tokens still fold into the lexical channel), kicks the warm-up, and returns `degraded: true` (in both `include:["tools"]` and `["extensions"]`) instead of blocking. Full semantic rankings (`degraded: false`) resume automatically once the model is ready. This mirrors `/api/search/messages`'s degraded/keyword fallback (SRCH-08).

### Whole-extension chips (embeddings, in-process)

The tool surface is already registry-wide (every enabled extension's tools), so a tool chip proposes *a tool*. An **extension chip** proposes bolting on *a whole unwired extension* whose intent spans the draft — e.g. "help me clean up my downloads folder" surfaces 🧩 `file-organizer` even though none of its tools are wired yet.

1. The same fast request rides `include:["tools","extensions"]` (one round-trip). The route ranks extensions only when the request asks for them.
2. **Candidates** = enabled extensions **minus** the ones already wired into the conversation (matched on extension **IDs**, `resolveSuggestableExtensions` in `scoped-tools.ts`), keeping only those with a non-empty manifest description. This is a fail-closed read: it runs only after `resolveScopedTools` has confirmed conversation ownership (unowned/missing → 404), never on its own.
3. **Suppressed inside a curating mode.** `isModeToolRestricted(mode)` is true when the mode pins `extensionIds` or carries a *narrowing* `toolRestriction` (`read-only`/`none`/`allowlist`) — the mode already decides which extensions belong, so extension chips are skipped (and the DB isn't even queried) in that case. `toolRestriction: "all"` is the stored default on every mode row and means "no filter", so plain modes keep the registry-wide surface open and extension chips stay meaningful inside them.
4. **Ranking** reuses the SAME draft embedding + priors computed once for the tool half (`rankScopedExtensions`): cosine of the draft against the extension's name+description embedding, folded as a MAX with each authored extension-level `suggestExamples` phrasing (embedded verbatim, query↔example), plus the lexical token-overlap signal. The per-extension prior is the **max** over the extension's `${name}__`-prefixed tool priors (`deriveExtensionPriors`) — so an extension the user leans on ranks higher without a many-tool extension over-counting.
5. **Higher bar, fewer slots.** `EXTENSION_SUGGEST_DEFAULTS` = top-**2**, minScore **0.35** (deliberately above the 0.32 noise cosine measured live) — an extension chip commits the user to a heavier accept than a single tool, so the relevance bar stays clear of semantic noise.
6. **Never both.** Any extension already represented by a ranked tool chip is deduped out of the extension list server-side, so one extension never yields a tool chip *and* an extension chip in the same response.
7. Clicking 🧩 wires `![ext:<name>]` and opens the extension's inline-tool UI with no preselection (1 tool → form, >1 → picker, 0 tools → nothing; the pill still wired the extension). Extension chips use a distinct violet palette vs the indigo tool chips.

The enhancement half is **not** fed extension candidates in v1 — a rewrite would lean on tools that aren't wired yet.

### Enhancement half (local LLM, optional sidecar)

1. The same pause fires a second request (`include:["enhance"]`) so chips never wait on the model. Skipped when the draft carries mention chips (a rewrite would clobber them) or while the 5-minute sidecar-unavailable backoff holds.
2. `src/suggest/enhance.ts` calls the configured OpenAI-compatible endpoint (`/v1/chat/completions`) with a JSON-schema `response_format` (retrying once without it for servers that reject the param), temperature 0.3, `/no_think` + defensive `<think>` stripping, 12 s timeout, and a TTL-cached availability probe. Every failure path returns null — the row simply doesn't render.
3. The rewrite renders with **Apply** (never auto-applies; keeps intent/language per the system prompt) and flips to **Undo** after applying. Stale responses are dropped via a normalized draft-key guard, and in-flight requests abort on further typing — no answer for a prompt that no longer exists.

### Configuration

| Key | Default | Meaning |
|---|---|---|
| setting `suggest:enabled` | `true` | **Global override** (Settings → Personalization → Composer suggestions): off ⇒ suggestions off in every project; on ⇒ each project's own toggle governs. |
| setting `project:<id>:suggest:enabled` | `true` | **Per-project toggle** (Project → Settings → Composer suggestions). Mirrors the `project:<id>:systemPrompt` key convention. Server-side, the scoping conversation's own project is authoritative; a client-supplied `projectId` only covers calls with no conversation yet. |
| setting `suggest:ollama-url` / env `EZCORP_SUGGEST_OLLAMA_URL` | compose: `http://ollama:11434` (prod) / `http://localhost:11434` (dev) | Local model endpoint. Unset ⇒ enhancement off, chips unaffected. |
| setting `suggest:model` / env `EZCORP_SUGGEST_MODEL` | `qwen3:1.7b` | Ollama tag. GPU tier: `qwen3:4b`. `ollama-init` pulls whatever is set. |

Settings win over env vars. The compose sidecar follows the SearXNG contract: always-on, hard resource caps, no `depends_on`, delete the `ollama`/`ollama-init` services to run without it.

**Consent contract (binding):** suggestions never modify the draft on their own. Tool chips insert a mention only when clicked; the ✨ rewrite renders as a preview and replaces the draft only on an explicit **Apply** (always followed by an **Undo** affordance). There is no auto-apply path.

### Telemetry (and its privacy contract)

`POST /api/composer/suggest/feedback` records `{kind: tool|enhance|extension, action: shown|accepted|dismissed, toolName?, latencyMs?}` into `suggestion_feedback` — **never draft text** (the zod schema is `.strict()`, so a draft field is rejected outright). The `extension` kind carries the extension name in `toolName`; a `shown` impression is posted per non-empty list (tool and/or extension) under a single per-draft guard, and acceptance is the watchdog that keeps authored-example ranking honest. Acceptance-rate rollups ride the `(kind, action, created_at)` index; this is the data that decides whether the enhancement half keeps its sidecar. Deleting a user cascades their telemetry away.

**Privacy, plainly:** what you type is analyzed in memory on your own server and discarded; nothing you type is stored by this feature; nothing is used to train any model unless the operator explicitly runs the offline export below.

## Fine-tuning on your own data (offline, optional)

The base model works cold. When an install has real usage, the operator can fine-tune the suggestion model on the platform's own prompt→tool history — entirely offline, on data that never leaves their control:

1. **Export** (on the EZCorp host): `bun scripts/suggest/export-training-data.ts [--days 365] [--out .ezcorp/suggest-training/dataset.jsonl]`. Pairs every successful tool call with the user message that triggered it (`src/suggest/training-export.ts`) and emits chat-format JSONL (`system` = tool-selection instruction, `user` = prompt, `assistant` = `{"tools":[…]}`). A few hundred clean examples is the point where a LoRA starts beating the base model on this narrow task.
2. **Train** (any GPU box or a free Colab T4, with [Unsloth](https://unsloth.ai/docs)): load `unsloth/Qwen3-1.7B` in 4-bit, attach a LoRA, train on the JSONL with the **same chat template** as inference, then `model.save_pretrained_gguf("out", tokenizer, quantization_method="q4_k_m")`.
3. **Serve:** copy the GGUF back, `ollama create ezcorp-suggest -f Modelfile` (a two-line Modelfile: `FROM ./out.gguf`), set `EZCORP_SUGGEST_MODEL=ezcorp-suggest`, restart. The runtime picks it up through the normal config path — no code changes.

## Design decisions & alternatives rejected

- **Fine-tuned 1B/4B doing both halves (the original framing):** rejected — LLM-as-router is the slow/expensive shape, a fine-tune has nothing to train on at install time, and 4B is CPU-hostile. The per-user retrieval prior delivers the personalization the "train on our data" ask actually wants, day one; the LoRA path above remains for installs that outgrow it.
- **Embeddings-only:** rejected — can't generate rewrites.
- **Default OFF:** review panel argued for it (Clippy risk); shipped ON per product owner's platform-wide yolo-default convention, with the mitigations that make ON tolerable: relevance threshold + min-draft gate (no popover without something relevant to say), per-draft Esc dismiss, one-click platform kill switch, and acceptance telemetry to re-litigate with data.
- **Suggestions in `PanelChatInput` (Ez panel):** deferred — the chat page is the target surface; the panel composer stays clean.
